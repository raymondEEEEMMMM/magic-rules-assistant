## 数据层设计

### 1. 规则数据 (SQLite)

**位置**: `magic_rules.db` (368KB)

| 表名 | 记录数 | 用途 |
|------|--------|------|
| `rules` | 1042条 | 万智牌综合规则 |
| `keyword_abilities` | 190条 | 关键词异能 |
| `card_rules` | 0条 | 卡牌特定规则（待导入） |

**访问方式**:
- 云函数启动时从云存储下载到 `/tmp/magic_rules.db`
- 利用 `/tmp` 目录的持久性（同一实例多次请求共享）
- 冷启动后无需重复下载

### 2. 卡牌数据 (外部 API)

**主要来源**: MTGCH API

| 数据源 | 用途 | 访问方式 |
|--------|------|---------|
| MTGCH API | 卡牌搜索、详情 | HTTP 请求 |
| AllPrintings.sqlite.gz | 离线卡牌查询（可选） | 云存储下载 |

**数据文件**:
- `AllPrintings.sqlite.gz` (174MB) - 完整卡牌数据库
- `Keywords.json` (4KB) - 关键词列表
- `SetList.json` (8.7MB) - 扩展包列表

## 部署策略

### 关键设计原则

1. **代码与数据分离**
   - 代码包仅包含 Python 源码（约 160KB）
   - 大数据文件存储在云存储
   - 运行时按需下载

2. **缓存优化**
   - 利用 `/tmp` 目录持久性
   - 同一实例多次请求共享数据库
   - 冷启动后无需重复下载

3. **外部 API 集成**
   - 卡牌数据优先使用 MTGCH API
   - 大型数据库按需下载到 `/tmp`

## 数据流程

### 规则搜索流程

```
1. 客户端请求 → GET /api/search?q=combat
                  ↓
2. API 网关路由 → 云函数 magic-rules-api
                  ↓
3. 检查 /tmp/magic_rules.db 存在性
   ├─ 存在 → 使用缓存
   └─ 不存在 → 从云存储下载 (首次，~5-10秒)
                  ↓
4. 执行 SQLite 查询 → 返回匹配规则
                  ↓
5. 返回 JSON 响应
```

### 卡牌搜索流程

```
1. 客户端请求 → GET /api/mtgch/search?q=Lightning
                  ↓
2. API 网关路由 → 云函数 magic-rules-api
                  ↓
3. 调用 MTGCH API → HTTP 请求到 mtgch.cn
                  ↓
4. 解析响应 → 提取卡牌信息
                  ↓
5. 返回 JSON 响应
```

## 性能优化

### 1. 缓存策略

| 数据类型 | 缓存位置 | 缓存时长 | 更新方式 |
|---------|----------|----------|---------|
| 规则数据库 | `/tmp/magic_rules.db` | 实例生命周期 | 重新上传云存储 |
| 卡牌查询 | 无 | 每次实时查询 | MTGCH API |

### 2. 并发控制

```python
# 函数配置

```

### 3. 超时设置

```python
# 函数配置

```

### 3. 故障排查

| 问题 | 可能原因 | 解决方案 |
|------|---------|---------|
| `unable to open database file` | 数据库未下载或云存储链接过期 | 检查云存储链接，更新 CLOUD_STORAGE_URL |
| 首次请求超慢 | 冷启动 + 数据库下载 | 正常现象，约 5-10 秒 |
| API 返回 500 | 代码错误 | 查看函数日志 |

## 小程序集成

### API 调用封装

```javascript
// miniprogram/app.js
App({
  globalData: {
    apiBase: 'https://magic-rules-assistant-0a1904c329-1410769303.ap-shanghai.app.tcloudbase.com'
  },

  requestApi(endpoint, data = {}, method = 'GET') {
    const url = `${this.globalData.apiBase}${endpoint}`
    return new Promise((resolve, reject) => {
      wx.request({
        url,
        method,
        data,
        success: res => {
          if (res.statusCode === 200) {
            resolve(res.data)
          } else {
            reject(new Error(`请求失败: ${res.statusCode}`))
          }
        },
        fail: reject
      })
    })
  }
})
```

### 使用示例

```javascript
// 搜索规则
app.requestApi('/api/search', { q: 'combat' })
  .then(res => {
    console.log('规则搜索结果:', res)
  })

// 查询关键词
app.requestApi('/api/keyword', { k: 'Flying' })
  .then(res => {
    console.log('关键词:', res.result)
  })

// 搜索卡牌
app.requestApi('/api/mtgch/search', { q: 'Lightning', page_size: 10 })
  .then(res => {
    console.log('卡牌列表:', res.items)
  })
```

## 总结

本架构的优势：

1. **轻量部署**: 代码包仅 160KB，快速部署
2. **数据分离**: 大文件存储在云存储，按需加载
3. **高效缓存**: 利用 `/tmp` 目录持久性，避免重复下载
4. **外部集成**: 卡牌数据使用 MTGCH API，无需本地存储大数据库
5. **易于维护**: 数据更新只需上传云存储，无需重新部署函数

通过这种架构设计，项目可以实现快速部署、高效运行和便捷维护。
